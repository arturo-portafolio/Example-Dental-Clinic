Plataforma destino: Replit (desarrollo) + Vercel (despliegue, sin problemas de compatibilidad)

Objetivo
- Construir una aplicación web moderna, rápida y responsive (contenido SOLO en inglés) para la clínica dental ficticia “BrightSmile Dental” en “Lakeshore Haven, Aurora State”.
- IMPORTANTE: No usar NINGUNA base de datos. No conectar NADA a Replit ni a ningún motor de base de datos.
- Incluir:
  - Front-end completo (páginas/sections: Home, Services, Before & After, Meet the Team, Promotions, FAQ, Contact, Google Reviews).
  - Header completamente fijo (“sticky”) pegado arriba, que NO se desplace con el scroll.
  - Botón flotante estilo WhatsApp.
  - Local SEO (titles/meta, NAP consistente, schema JSON-LD tipo Dentist/LocalBusiness).
  - Estructura de “server”/API (archivos de backend) que HOY solo devuelven datos estáticos en memoria, sin persistencia, pero con arquitectura clara para que en el futuro se pueda conectar una base de datos si el desarrollador lo desea.
  - Panel de administración (UI + rutas API) que funcione en modo DEMO: estructura completa, pero sin persistir cambios (mock en memoria), claramente preparado para conectar una base de datos más adelante.

Alcance (MVP sin base de datos)
- Sitio público completamente funcional con contenido estático inicial y navegación.
- Admin panel (UI) funcional a nivel de interfaz y llamadas a API, pero:
  - Sin ningún tipo de almacenamiento persistente (NO DB, NO archivos de disco).
  - Respuestas de las rutas admin simuladas (mock) usando estructuras en memoria y/o constantes estáticas.
- API/“server” implementado vía rutas de API (por ejemplo Next.js /app/api o /pages/api), retornando datos estáticos de arrays/objetos locales.
- Código y estructura totalmente compatibles con despliegue en Vercel (sin servidores personalizados, sin dependencias incompatibles).

Restricciones críticas (muy importante)
- PROHIBIDO:
  - Usar Replit DB.
  - Usar SQLite, Postgres, MySQL, MongoDB, Supabase, Prisma, etc.
  - Leer o escribir datos a archivos para persistir contenido (el filesystem en Vercel es efímero).
  - Cualquier tipo de persistencia de datos en el backend.
- PERMITIDO:
  - Datos estáticos definidos en archivos JS/TS (por ejemplo /data/staticData.ts).
  - Mock de “repositorios” o “services” que lean de esos datos estáticos.
  - Estructura y comentarios tipo TODO para que en el futuro se pueda enchufar una DB real, pero sin implementar la conexión ahora.

Endpoints API (modo estático/mock)
- GET /api/services
  - Devuelve lista de servicios desde /data/services.ts.
- GET /api/promotions
  - Devuelve promociones “activas” según flags y fechas definidas ESTÁTICAMENTE en /data/promotions.ts.
- GET /api/team
  - Devuelve miembros activos desde /data/team.ts.
- GET /api/gallery
  - Devuelve items desde /data/gallery.ts, con filtro opcional por ?category.
- GET /api/faq
  - Devuelve preguntas desde /data/faq.ts.
- GET /api/settings/public
  - Devuelve NAP, horarios y configuración pública desde /data/settings.ts.
- POST /api/contact
  - Valida el body.
  - Sin DB: no guarda nada, puede imprimir en consola.
  - Devuelve { success: true }.
  - Incluir comentarios TODO para integración futura con servicio de email.
- GET /api/google-reviews
  - Si GOOGLE_API_KEY y PLACE_ID están definidos: llama a Places API y devuelve reviews.
  - Si no: devuelve un error controlado (ej. { error: "Not configured" }).

  - Compara email/password con variables de entorno (por ejemplo ADMIN_EMAIL, ADMIN_PASSWORD).

  - Invalida la cookie/token (en memoria o expira).
- GET /api/admin/me
  - Devuelve datos básicos del admin si el token/cookie es válido.

  - GET: devuelve datos desde /data/**.
